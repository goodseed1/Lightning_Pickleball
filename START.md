## **3. '자동화된 안전망': 정의 및 사용법 (가장 중요한 부분)**

'자동화된 안전망'(`npm test`)은 우리 프로젝트의 품질을 보증하는 **가장 중요한 시스템**입니다. 이것은 단순한 검사 도구가 아니라, **앞으로의 모든 개발 과정을 안내하는 '나침반'이자 '방패'**입니다.

**다음 AI 어시스턴트는 아래의 '골든 룰(Golden Rules)'을 반드시 따라야 합니다.**

##### **규칙 1: 새로운 기능 개발 시 - '테스트 주도 개발(TDD)'을 따르십시오.**

- **절차:**
  1.  **실패하는 테스트부터 작성하십시오:** 새로운 기능(예: '선수 랭킹 시스템')의 실제 코드를 작성하기 **전에**, `rankingService.test.ts` 파일에 **"이 기능은 이렇게 작동해야 한다"**는 기대를 담은 **실패하는 테스트 케이스**부터 만드십시오.
  2.  **테스트를 통과하는 코드만 작성하십시오:** 당신의 유일한 목표는, 이 실패하는 테스트를 '성공(PASS)'으로 바꾸는 **가장 단순하고 명확한 실제 코드**를 작성하는 것입니다.
- **왜?:** 이 방식은 당신이 항상 명확한 목표를 가지고 코드를 작성하게 만들며, 자연스럽게 더 좋은 설계를 유도합니다.

##### **규칙 2: 기존 기능 수정 시 - '회귀 테스트'를 수행하십시오.**

- **절차:**
  1.  코드를 수정하기 **전에**, `npm test`를 실행하여 현재 모든 것이 정상임을 확인하십시오.
  2.  수정을 완료한 후, **반드시 다시 `npm test`를 실행**하십시오.
  3.  **만약 단 하나의 테스트라도 실패한다면,** 당신의 수정이 의도치 않게 다른 기능을 망가뜨렸다는 뜻입니다. **그 코드를 절대 커밋해서는 안 됩니다.** 모든 테스트가 다시 통과할 때까지 문제를 해결하십시오.
- **왜?:** 이 규칙은 미래의 모든 개발자가 자신도 모르는 사이에 버그를 만드는 것을 원천적으로 방지하는, 우리 프로젝트의 가장 강력한 방패입니다.

##### **규칙 3: '안전망'을 먼저, 그리고 항상 신뢰하십시오.**

- 앱에서 원인을 알 수 없는 버그가 발생하면, 가장 먼저 **관련 '두뇌' 파일에 대한 단위 테스트를 추가**하여 문제 상황을 재현하십시오. 테스트가 실패하는 것을 확인한 뒤에야 비로소 실제 코드를 수정해야 합니다. 이것이 '추측'이 아닌 '증거'에 기반하여 버그를 잡는 가장 빠른 길입니다.

- 코드를 새로 만들기 전에 항상 전에 사용하던 코드가 있는지 확인해서 죽은 코드를 남기지 마십시오.

- 코드를 새로 만들거나 수정했을때 항상 다른 기능에 영향을 주지 않았나 점검하십시오.(예로 ESLint로 테스트함)
